# AI 简历优化系统 — 项目分析文档

## 一、项目概述

本项目是一个基于 **LangGraph 多智能体架构** 的 AI 简历优化与面试辅助系统。用户输入原始简历（PDF/Word）和目标岗位信息后，系统会自动完成简历解析、分模块优化、PDF/Word 排版渲染、自我介绍生成、面试题生成等全流程任务。

**核心价值**：针对不同技术岗位，利用大语言模型对简历的各个板块进行深度优化，同时生成个性化的面试准备材料。

---

## 二、技术栈

| 层面 | 技术 |
|------|------|
| **多智能体框架** | LangGraph 1.0 + LangChain 1.2 |
| **LLM 提供商** | 通义千问 (Qwen-Max) / DeepSeek / OpenAI 兼容协议 |
| **文档解析** | PyPDFLoader、Docx2txtLoader、pdfplumber、PyMuPDF |
| **PDF 生成** | ReportLab (Platypus) + 自定义中文字体 (SimHei) |
| **Word 生成** | python-docx |
| **CLI 界面** | Rich (进度条、面板、交互式输入) |
| **缓存系统** | 基于 SHA-256 的 JSON 文件缓存 |
| **配置管理** | python-dotenv + 集中配置文件 |

```
    └──────────┬──────────┘

    └──────────┬──────────┘
               │

```

**关键设计**：
- 模块优化智能体之间采用**串行链式**连接，因为 `self_evaluation` 需要依赖其他所有模块的优化结果
- `extractor` 之后有**条件分支**：提取失败则直接终止
- 每个模块智能体内部都集成了**记忆缓存**逻辑，命中缓存时跳过 LLM 调用

## 四、核心模块详解

### 4.1 智能体系统

#### 4.1.1 提取智能体 (Extractor)
- **职责**：扫描 `data/` 目录，自动识别 PDF/DOCX/DOC 简历文件并提取纯文本
- **工具**：`read_resume_tool`（LangChain @tool 装饰器，MCP 兼容）
- **输出**：原始简历文本 → 写入 `AgentState.original_resume_text`

#### 4.1.2 模块优化智能体 (Section Agents) × 8
- **通用框架**：`_run_section_agent()` 函数封装了统一的处理逻辑
- **处理流程**：
  1. 检查记忆缓存（SHA-256 签名匹配）
  2. 缓存命中 → 直接返回；未命中 → 构建 Prompt 调用 LLM
  3. 后处理（Markdown 清洗、格式规范化）
  4. 生成中间结果 PDF
  5. 更新 `sections` 字典

| 智能体 | 优化内容 | 特殊处理 |
|--------|----------|----------|
| title | 简历标题 | 无 |
| personal_info | 个人信息 | JSON 解析 → 强制覆盖求职意向、补全默认值 |
| skills | 专业技能 | 强制列表格式（`- ` 前缀） |
| certificate | 技能证书 | 列表格式保证 |
| work_experience | 职业经历 | 竖线分隔格式 `角色 \| 公司 \| 时间` |
| education | 教育经历 | 同上三列格式 |
| project_experience | 项目经验 | 复杂清洗：子标题/描述/技术栈/职责/亮点 分段解析 |
| self_evaluation | 自我评价 | 合并为单段落、去除换行 |

#### 4.1.3 组装器 (Assembler)
- 按**固定顺序**拼接所有模块：标题 → 个人信息 → 专业技能 → 技能证书 → 职业经历 → 教育经历 → 项目经验 → 自我评价
- 调用 `convert_markdown_to_pdf()` 生成最终 PDF
- 同时保存 Markdown 版本

#### 4.1.4 排版渲染智能体 (Layout Expert)
- 根据输入文件类型决定输出格式（PDF 或 Word）
- PDF 使用 ReportLab Platypus 引擎
- Word 使用 python-docx

#### 4.1.5 面试官智能体 (Interviewer)
- **自我介绍**：约 200 字，基于优化后的简历生成
- **面试题**：采用**分批并行**策略生成
  - 4 类题目：项目针对性 (10%)、基础技能 (30%)、进阶技能 (30%)、场景设计 (30%)
  - 使用 `ThreadPoolExecutor` 并发调用 LLM
  - 当前默认生成 1 道题（可配置）

### 4.2 状态管理 (AgentState)

```python
class AgentState(TypedDict):
    # 输入
    job_name: str                    # 目标岗位
    job_age: str                     # 工作年限
    technology_stack: str            # 技术栈配置
    work_dir: str                    # 工作目录

    # 中间数据
    original_resume_text: str        # 原始简历文本
    optimized_resume_text: str       # 优化后完整简历 (Markdown)
    sections: Dict[str, str]         # 各模块优化结果

    # 输出
    self_intro: str                  # 自我介绍
    interview_questions: str         # 面试题

    # 追踪
    memory_hits: Annotated[List[str], operator.add]  # 缓存命中记录
    progress: Annotated[List[str], operator.add]      # 进度记录
    logs: Annotated[List[str], operator.add]          # 详细日志
```

**设计特点**：`memory_hits`、`progress`、`logs` 使用 `Annotated[List, operator.add]` 实现追加式合并，LangGraph 自动处理并发写入。

### 4.3 LLM 工厂 (Factory)

支持两个供应商，通过 `.env` 中的 `LLM_PROVIDER` 切换：

| 供应商 | 环境变量 | 模型 |
|--------|----------|------|
| DashScope (通义) | `DASHSCOPE_API_KEY` | qwen-max |
| DeepSeek | `DEEPSEEK_API_KEY` + `DEEPSEEK_BASE_URL` | deepseek-chat (可配) |

DeepSeek 供应商使用自定义的 `OpenAIChatModel` 封装类，兼容 LangChain 的 `invoke()` 接口，统一了消息格式处理。

### 4.4 记忆/缓存系统 (Memory Manager)

- **缓存键**：`SHA-256(原始简历文本 + JSON(上下文))`
- **上下文因素**：`job_name` + `technology_stack`（影响优化结果，故纳入签名）
- **特殊处理**：`self_evaluation` 额外纳入 `sections_summary`（其他模块的汇总），因为自我评价依赖其他模块的内容
- **存储格式**：JSON 文件 `{section_key}_{hash}.json`
- **生命周期**：支持交互式管理（保存/覆盖/清理），在 `main.py` 中提供用户交互入口

### 4.5 技术栈路由 (Router)

**两级路由策略**：

1. **规则匹配**（优先）：基于岗位名称关键词的优先级规则链
   - 支持的岗位：Java后端、Python后端、前端、测试开发、测试、运维/SRE、产品经理、Java+LLM、Python+LLM 等
   - 特殊规则 > 通用规则（如 "Java大模型" > "Java"）

2. **语义匹配**（兜底）：使用 LLM 从 12 个技术栈文件中选择最匹配的
   - 现有技术栈文件：`python_backend.txt`、`java_backend.txt`、`frontend.txt`、`llm_app_dev.txt`、`devops.txt`、`sre.txt`、`testing_manual.txt`、`test_dev.txt`、`product_manager.txt`、`java_llm.txt`、`llm_testing.txt`、`llm_product_manager.txt`

### 4.6 PDF 生成引擎

基于 **ReportLab Platypus** 构建，针对简历场景做了大量定制：

| 板块 | 渲染策略 |
|------|----------|
| 标题 | 居中、30pt、加粗 |
| 个人信息 | 双列 Table 布局（50%/50%） |
| 职业/教育经历 | 三列 Table（角色35% / 公司40% / 时间25%），时间右对齐 |
| 专业技能 | 列表 + 大行距 (2.0x) |
| 项目经验 | Key-Value Table 布局（子标题不加粗） |
| 自我评价 | 首行缩进 + 两端对齐 |

**中文字体**：使用 SimHei，注册为普通和粗体两种变体。

## 五、数据流

```
用户输入 (岗位/年限) + data/ 目录中的简历文件
        │
        ▼
  ┌─ extractor ──→ 原始文本
  │
  ├─ route ──→ 匹配技术栈文件
  │
  ├─ 8× section_agents ──→ sections: Dict[str, str]
  │     │
  │     └─ 每个 agent: 检查缓存 → LLM 优化 → 后处理 → 中间 PDF
  │
  ├─ assembler ──→ optimized_resume.md + Optimized_Resume.pdf
  │
  ├─ layout_expert ──→ 最终 PDF 或 Word
  │
  └─ interviewer ──→ self_introduction.md + interview_questions.md
```

## 八、架构亮点

1. **多智能体协作**：LangGraph 串行图 + 条件分支，职责清晰、易于扩展
2. **记忆增强**：基于内容签名的智能缓存，避免重复 LLM 调用，节省成本
3. **两级路由**：规则匹配 + LLM 语义兜底，兼顾速度和灵活性
4. **多供应商 LLM**：工厂模式统一接口，一键切换供应商
5. **精细化 PDF 排版**：针对简历的每个板块设计了专门的渲染策略
6. **交互式记忆管理**：用户可选择保存/覆盖/清理缓存
7. **并发面试题生成**：ThreadPoolExecutor 分批并行，提升效率

## 九、可改进方向

### 9.1 架构层面
- **串行瓶颈**：8 个模块智能体目前是串行执行的（LangGraph 的边定义），其中 `self_evaluation` 必须串行，但其他 7 个模块理论上可以并行。可使用 LangGraph 的 fan-out/fan-in 模式优化执行时间
- **缺少错误恢复**：某个模块失败后整个流程终止，可引入重试机制或降级策略

### 9.2 功能层面
- **面试题数量偏少**：当前默认仅 1 道，可适当增加
- **缺少 Web UI**：当前仅 CLI 交互，可增加 Flask/FastAPI Web 界面
- **缺少版本管理**：优化前后对比功能
- **字体单一**：仅支持 SimHei，可扩展更多中文字体

### 9.3 工程层面
- **缺少单元测试**：`utils/` 目录下有测试脚本但非正式测试套件
- **缺少日志系统**：使用 Rich 控制台输出，建议引入标准 logging 模块
- **硬编码较多**：部分配置（如 PDF 列宽比例）分散在代码中，可集中到 `conf.py`
- **requirements.txt 包含平台依赖**：`pywin32` 仅 Windows 可用，应条件安装
