# 面试题集锦

## 项目针对性提问

**Q1: 在你的“RAG智能客服系统”项目中，提到了针对400万级海量历史数据的大规模并发跑批验证，并创新性设计了断点续跑机制。能否详细阐述一下这个跑批任务的整体架构设计？在基于Celery实现高并发异步处理时，你是如何具体实现“断点续跑”机制以保障数据完整性的？在这个高并发写入和状态更新的过程中有遇到哪些性能瓶颈，你是如何优化的？**

*详解:*

这是一道非常契合候选人简历亮点的综合型深度面试题，旨在考察候选人对于**分布式任务队列（Celery）的底层理解、大规模数据处理的状态管理能力，以及在高并发场景下的问题排查与调优经验**。优秀的候选人应该能够从架构设计、核心机制实现、以及性能调优三个维度来进行作答。

**1. 跑批任务的整体架构设计**
候选人应当首先说明为什么选择 Celery + Redis 的组合来处理400万级数据。传统的同步处理耗时极长且容易阻塞，而引入 Celery 可以实现业务的解耦和分布式并发计算。
*   **任务拆分:** 明确不应将400万条数据作为一个单一任务扔进队列，而是应该采用“分而治之”的思想。通过生产者（通常是一个独立的 Celery Task 或者 FastAPI 的接口触发），按批次（例如每次读取 5000 或 10000 条记录）从数据库或文件中读取数据。
*   **工作流编排:** 利用 Celery 的 `group` 或 `chord` 等原语将切分后的子任务分发给多个 Worker 节点并发执行问答对验证。
*   **结果汇聚:** 跑批完成后，需要统计整体指标（准确率、召回率、响应时间）。可以通过 Redis 原子计数器或者将中间结果写入 MongoDB/ES，最后由一个聚合任务统一计算生成报告。

**2. “断点续跑”机制的具体实现**
这是本题的核心考点，主要考察候选人对任务状态的持久化管理能力。普通的任务一旦重启或崩溃，中间状态即丢失。实现断点续跑，候选人可以按以下逻辑回答：
*   **全局任务状态记录表:** 在数据库（如 MySQL 或 Redis Hash）中设计一张专门的 `BatchJobRecord` 表。每次发起大规模跑批时，生成一个唯一的 `batch_id`（跑批编号，简历中也提到了用于溯源）。
*   **进度游标记录:** 记录当前这个 `batch_id` 已经处理到的游标位置（例如最后一条数据的 ID 或偏移量 offset），以及当前成功的数量、失败的数量。
*   **原子性保障:** 每个子任务（或小批次任务）处理完毕并成功写入验证结果后，通过 Redis 的 `INCRBY` 或者数据库的行级锁更新进度游标。
*   **异常捕获与恢复:** 当系统因 OOM、宕机或重启导致任务中断时，系统不会自动盲目重跑。用户可以通过输入 `batch_id` 发起“续跑”操作。续跑服务的逻辑是：查询该 `batch_id` 的状态记录，获取上次断开的游标位置，然后从该游标位置继续向 Celery 队列投递后续的数据切片任务。

**3. 遇到的性能瓶颈与优化方案**
高并发场景下处理大吞吐量必定会遇到问题，候选人需要展示其实战经验：
*   **瓶颈 1：数据库/ES 读写 I/O 瓶颈（连接池被打满）**
    *   *现象:* 400万并发请求导致后端存储连接数飙升，出现超时。
    *   *优化方案:* 在 FastAPI/Celery 中配置合理的数据库/ES连接池，并控制 Celery 的并发度（如通过 `--concurrency` 参数）。将部分写入操作由“同步单条写入”改为“异步小批量 Bulk 写入”（例如使用 Elasticsearch 的 Bulk API 或 Django的 `bulk_create`），大幅降低 I/O 压力。
*   **瓶颈 2：大文件/大结果集导致的 OOM（内存溢出）**
    *   *现象:* 一次性把几百万条数据加载到内存中进行切片导致内存溢出。
    *   *优化方案:* 使用生成器或者基于游标的分页（`id > last_id limit 1000`）来流式读取数据，避免内存堆积。
*   **瓶颈 3：Celery 队列堵塞与消息丢失风险**
    *   *现象:* 400万消息瞬间涌入 Redis/RabbitMQ 导致队列积压，甚至内存撑爆。
    *   *优化方案:* 调整预取值 `worker_prefetch_multiplier`，控制每个 Worker 获取的任务数，防止个别 Worker 拿太多任务导致其他 Worker 空闲或任务分配不均；在 Celery 配置中开启 ACKs Late（任务执行完才确认），配合死信队列处理多次重试仍然失败的脏数据，确保数据不丢失。
