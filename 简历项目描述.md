## 版本一：精简版（适合简历空间有限）

### AI 智能简历优化系统

**技术栈**: Vue 3 + TypeScript + Tailwind CSS + FastAPI + LangGraph + WebSocket + ReportLab

- 主导全栈架构设计，前端 Vue 3 + TypeScript，后端 FastAPI + LangGraph 多 Agent 工作流，实现前后端分离 + WebSocket 实时通信
- 设计 WebSocket 消息协议，后端 AI Agent 每完成一个节点通过 asyncio.Queue 桥接推送到前端，替代轮询方案，用户可实时看到 12 步优化进度
- 前端封装 `useWebSocket` Composable 管理连接生命周期，开发 AgentPipeline 进度可视化（5 种状态动态切换）、拖拽上传等核心组件
- 后端实现 REST API（文件上传/下载/缓存管理）+ WebSocket 优化流端点，设计会话管理和多格式文件输出
- 集成 LangGraph 多 Agent 流水线（12 个专职 Agent），设计 Memory 缓存机制减少 60%+ 重复 LLM 调用

## 版本二：标准版（推荐）

### AI 智能简历优化系统

**技术栈**: Vue 3 + TypeScript + Tailwind CSS + Pinia + FastAPI + WebSocket + LangGraph + ReportLab

**项目简介**:
基于 AI 多 Agent 架构的全栈简历优化平台。用户上传简历后，系统通过 12 个专职 Agent 流水线式优化各模块，输出 PDF 简历、自我介绍及面试题。独立负责前后端开发。

**主要工作**:

**前端（Vue 3 + TypeScript）**:

- 基于 Vue 3 Composition API 搭建项目，TypeScript 实现从 API 层到组件层的全量类型覆盖
- 设计 WebSocket 实时通信方案：封装 `useWebSocket` Composable，按消息类型（status/agent_done/progress/complete/error）分发状态，组件卸载自动清理连接
- 开发 AgentPipeline 进度可视化组件：12 个 Agent 节点 5 种状态动态追踪，进度实时计算
- 实现 FileUpload 拖拽上传（HTML5 Drag & Drop + FormData）、ResultCard 多格式下载等核心组件
- Pinia 状态管理 + Vue Router 三页面流转；Vite 开发代理解决跨域

**后端（FastAPI + LangGraph）**:

- 使用 FastAPI 搭建 REST API + WebSocket 服务，实现文件上传、会话管理、技术栈查询、文件下载等接口
- 设计 WebSocket 优化流端点：asyncio.Queue 桥接同步 Agent 执行与异步 WebSocket 推送，实时推送 12 个节点的执行状态
- 基于 LangGraph 状态图编排 12 个 Agent 节点，通过 TypedDict AgentState 在节点间自动流转状态
- 设计 Memory 缓存系统：SHA-256 上下文感知哈希，相同简历二次优化跳过 LLM 调用，节省 60%+ token 成本
- 实现多 LLM 适配层（通义千问/DeepSeek/智谱GLM），工厂模式 + 统一 ChatModel 接口，改配置即可切换模型

**项目亮点**:

- **全栈实时通信**：WebSocket + asyncio.Queue 实现后端同步 Agent → 前端异步 UI 的实时进度推送
- **成本优化**：Memory 缓存 + 规则优先路由，减少非必要 LLM 调用
- **架构可扩展**：新增简历模块只需注册一个 Agent 节点 + 一个 Prompt，零改动现有流程

## 版本三：详细版（适合面试口头介绍或作品集）

### AI 智能简历优化系统

**技术栈**: Vue 3 + TypeScript + Tailwind CSS 4 + Pinia 3 + FastAPI + WebSocket + LangGraph + LangChain + ReportLab + Vite 6

**项目背景**:
基于 AI 多 Agent 架构的全栈简历优化平台。用户上传简历并输入目标职位后，系统通过 12 个专职 AI Agent 流水线式优化简历各模块，最终输出 PDF 优化简历、自我介绍和面试题。独立负责前后端架构设计与开发。

**架构设计**:

```
前端 (Vue 3 + TS)          后端 (FastAPI)              AI 引擎 (LangGraph)
┌──────────────┐     REST     ┌──────────┐          ┌──────────────────┐
│ Home         │ ───────────→ │ Upload   │          │ Extractor Agent  │
│ Optimize     │   WebSocket  │ Download │   ────→  │ 8x Section Agent │
│ Result       │ ←─────────── │ Memory   │          │ Assembler Agent  │
└──────────────┘              └──────────┘          │ Layout Agent     │
                                                    │ Interviewer Agent│
                                                    └──────────────────┘
```

**前端核心实现**:

1. **WebSocket 实时通信**: 封装 `useWebSocket` Composable，建立长连接后按 message type 分发到响应式状态，组件卸载时自动 close 防止内存泄漏
2. **AgentPipeline 可视化**: 12 个节点纵向排列，Clock/Loader/CheckCircle/Zap/AlertCircle 五种状态动态切换，进度百分比实时计算
3. **拖拽上传**: HTML5 Drag & Drop API + FormData，拖拽悬停反馈、格式校验、状态展示
4. **TypeScript 类型体系**: 联合类型定义 WebSocket 消息、接口类型定义 API 响应、类型收窄保障消息处理安全

**后端核心实现**:

1. **FastAPI 全双工服务**: REST API（上传/下载/缓存管理）+ WebSocket 优化流端点，会话管理 + CORS 配置 + 静态文件托管
2. **WebSocket 推送桥接**: 后端 Agent 同步执行 → asyncio.Queue → 异步消费推送到前端，解决同步/异步模型不匹配问题
3. **LangGraph 工作流**: StateGraph 编排 12 个 Agent 节点，TypedDict AgentState 自动流转，条件路由处理异常分支
4. **Memory 缓存**: SHA-256(content + job + techStack) 作为缓存 key，上下文感知去重，相同简历二次优化大部分节点直接命中缓存
5. **多 LLM 适配**: 工厂模式封装通义千问/DeepSeek/智谱GLM，统一 ChatModel 接口，环境变量切换模型
6. **PDF 生成**: ReportLab + TTF 中文字体注册，多列布局、表格样式、自定义间距，输出专业排版 PDF

## 面试高频问题 & 参考回答

### Q: "你说独立开发，Python 也是你自己写的？"

> "对，后端 API 层和 WebSocket 这部分是我写的。LangGraph 的 Agent 工作流部分我参考了 LangChain 的文档和示例，边学边做的。Python 语法上遇到问题就查文档，核心的架构设计、数据流、API 设计都是我自己决定的。"

### Q: "为什么选 FastAPI 不选 Express？"

> "因为 AI 部分用了 Python 的 LangChain/LangGraph 生态，后端必须用 Python。在 Python Web 框架里，FastAPI 天然支持 async 和 WebSocket，自带 OpenAPI 文档，性能也接近 Go 的并发水平，比 Flask/Django 更适合这种场景。"

### Q: "Memory 缓存的 key 为什么用 SHA-256 哈希？"

> "因为同一个简历投不同职位，优化结果应该不同。所以缓存 key 包含了原始内容、职位名称和技术栈三个维度，哈希后做文件名。命中缓存就直接返回，跳过 LLM 调用。实际使用中，微调简历的时候大部分 section 不会变，缓存命中率很高。"

### Q: "WebSocket 的 asyncio.Queue 桥接是怎么工作的？"

> "LangGraph 的 `graph.stream()` 是同步阻塞的，但 WebSocket 发送需要异步。所以我用 asyncio.Queue 做桥梁：Agent 在线程中同步执行，每完成一个节点就 `put_nowait` 一个事件到队列；另一边异步协程从队列 `get` 出来通过 WebSocket 推给前端。用 `run_in_executor` 把同步图执行放到线程池里，和异步 WebSocket 并行跑。"

### Q: "前端 TypeScript 类型安全具体体现在哪？"

> "WebSocket 消息我用联合类型 `StatusMsg | AgentDoneMsg | ProgressMsg | CompleteMsg | ErrorMsg`，每个类型有不同的字段。处理消息时先判断 type 做类型收窄，TypeScript 会自动推断出对应类型的字段。API 层也定义了 UploadResponse、TechStacksResponse 这些接口，请求返回值有类型提示。"

### Q: "多 LLM 适配是怎么做的？"

> "工厂模式。`get_llm()` 函数检测环境变量里配了哪个 API Key，返回对应的 ChatModel 实例。三个模型都兼容 OpenAI 的接口格式，所以用 LangChain 的 ChatOpenAI 做统一封装，对外暴露的接口完全一样。切模型只改 .env 里的 Key，代码不用动。"

### Q: "这个项目的难点在哪？"

> "主要两个。一个是 WebSocket 的同步/异步桥接，LangGraph 同步执行但前端需要异步接收，中间的 asyncio.Queue 这层如果设计不好会出现事件丢失或连接超时。另一个是 PDF 排版，中文简历有字体、间距、多列布局这些需求，用 ReportLab 从零调样式比较耗时，调试了好几轮。"
